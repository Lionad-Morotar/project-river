# packages/pipeline Code Review

**Phase:** 07.1-02
**Scope:** `packages/pipeline` — parser, calcDay, sumDay, analyze, CLI, tests, configuration
**Date:** 2026-04-09

## Summary

`packages/pipeline` implements the core Git log parser, daily aggregation logic, cumulative stat generator, CLI entrypoint, and database persistence layer. The code is functionally correct and well-tested where a database is available, but it currently fails lint (36 errors, 1 warning) due to import/export sorting, quote-props inconsistency, member-delimiter style, and chaining formatting. Test coverage is strong for parser and calcDay, but integration tests for DB-dependent paths (`sumDay`, `analyze`) silently skip when `DATABASE_URL` is absent.

### Style Observations
- `src/cli.ts` lines 10-12: object keys `'batch-size'`, `'force'`, `'incremental'` use mixed quoting (`'batch-size'` is required because of the hyphen; `force` and `incremental` lack quotes). `antfu/eslint-config` enforces `style/quote-props` consistency.
- `src/db/analyze.ts` line 82: `{ id: number; hash: string }[]` uses semicolons as type member delimiters instead of commas. Line 103: same issue in `{ commitId: number; path: string; insertions: number; deletions: number }[]`.
- `tests/analyze.test.ts` line 78-79: `db.select().from(commit_files)\n.where(...)` breaks the `antfu/consistent-chaining` rule.
- `tests/sumDay.test.ts` lines 77-85: query builders are split across lines between `.where()` and `.orderBy()`, triggering the same chaining rule.

## Type Safety

### Exported Types
- `src/types.ts` exports `FileChange` and `ParsedCommit` interfaces. These are clear, well-named, and used consistently across the package.
- `src/calcDay.ts` re-exports its own `DailyStat` interface (line 3), which is the primary consumer-facing type for aggregation results.

### Function Signatures
- `parseRepo(repoPath: string): AsyncGenerator<ParsedCommit>` — correct.
- `parseLogStream(lines: AsyncIterable<string>): AsyncGenerator<ParsedCommit>` — correct.
- `calcDay(commits: ParsedCommit[]): DailyStat[]` — correct.
- `generateSumDay(projectId: number): Promise<void>` — correct.
- `analyzeRepo(repoPath: string, projectName: string | undefined, options: AnalyzeOptions): Promise<void>` — correct, though `projectName` could be typed as optional (`projectName?: string`) rather than `string | undefined` for ergonomics.

### Any Usage
- No explicit `any` types were found in the source files.
- `parseArgs` in `src/cli.ts` (line 7) returns `values['batch-size'] as string` and `values.force as boolean` (lines 22, 29, 30). These casts are unnecessary because narrowing `parseArgs` with a typed `options` declaration would give proper inference. This is a minor type-safety erosion.

### Type Assertions
- `src/db/analyze.ts` line 42: `inserted[0]!.id` uses a non-null assertion. Same pattern on line 49 (`inserted[0]!.id`). In both cases the array is guaranteed to have exactly one element because `.returning()` returns the inserted row, but explicit checks or safer unwraps would be preferable.

## Test Coverage

### Unit Tests (no DB required)
- `tests/parser.test.ts` (5 tests)
  - Covers `parseLogStream` basic parsing, binary file mapping (`-` → 0), empty commits, and `parseRepo` integration with `.mailmap` resolution.
  - Also tests invalid repo path error handling.
- `tests/calcDay.test.ts` (4 tests)
  - Covers basic aggregation, UTC day boundary with negative timezone offsets, multi-contributor/multi-day separation, and empty commit handling.
- `tests/cli.test.ts` (3 tests)
  - Uses `vi.doMock` to isolate `runAnalyze` from `analyzeRepo`. Tests defaults, explicit flags, and missing positional argument error.

### Integration Tests (require DB)
- `tests/sumDay.test.ts` (2 tests)
  - Skipped entirely when `DATABASE_URL` is absent with a `console.warn`.
  - Tests single-contributor cumulative stats and multi-contributor isolation.
- `tests/analyze.test.ts` (3 tests)
  - Skipped when `DATABASE_URL` is absent.
  - Tests `force: true` across months, duplicate-analysis rejection, and incremental SHA skipping.

### Mocking Strategy
- `cli.test.ts` correctly intercepts `analyzeRepo` with `vi.doMock` before importing `cli.ts`, allowing the CLI to be tested without a live DB.
- `parser.test.ts` builds real temporary Git repositories via `execSync('git init')`, which is an acceptable lightweight integration approach.
- `analyze.test.ts` also creates real temp Git repos; combined with a live DB connection, this provides end-to-end confidence but requires environment setup.

### Gaps / Observations
- No unit tests for `generateSumDay` without a DB. The logic is a single CTE SQL query; a mocked DB test is not present but may not be critical.
- `analyzeRepo` has no unit tests for its branching logic (force vs incremental vs new project) outside of the DB integration tests.
- Edge case: `parseRepo` does not handle `git log` stderr or stdout parse errors beyond the exit-code check on line 22-25. There is no test for malformed log lines.

## Architectural Consistency

### Package Exports (`src/index.ts`)
```ts
export * from './calcDay.ts'
export * from './parser.ts'
export * from './types.ts'
export * from './db/sumDay.ts'
export * from './db/analyze.ts'
```
- The `package.json` also defines conditional exports:
```json
"exports": {
  ".": "./src/index.ts",
  "./parser": "./src/parser.ts",
  "./calcDay": "./src/calcDay.ts"
}
```
- This is consistent. Library consumers can import `@project-river/pipeline` (root) or deep-import parser/calcDay. CLI code (`src/cli.ts`) lives in the same package but is not re-exported from `index.ts`, which is appropriate.

### Monorepo Alias Resolution
- `packages/pipeline/tsconfig.json` defines `paths` for `@project-river/db/client` and `@project-river/db/schema`.
- `packages/pipeline/vitest.config.ts` replicates the same aliases via `resolve.alias`.
- This matches the Phase 4 decision: "Use absolute path aliases in pipeline vitest.config.ts to resolve @project-river/db without relying on pnpm workspace links."

### Separation of CLI from Library Code
- `src/cli.ts` is thin (~41 lines). It performs argument parsing and delegates to `analyzeRepo`. This is a good separation of concerns.
- The shebang `#!/usr/bin/env bun` and `if (import.meta.main)` guard are appropriate for a Bun-targeted CLI.

### Configuration
- `vitest.config.ts` is minimal but sufficient.
- `tsconfig.json` uses `NodeNext` module resolution, which aligns with the project’s `"type": "module"` setup.
- `package.json` lacks `devDependencies` entries for `vitest` and `@antfu/eslint-config`. These are likely hoisted to the workspace root, which is acceptable in a pnpm monorepo.

## Findings

### Critical

1. **Lint failures block CI** — `pnpm lint` exits with 36 errors. 35 errors are auto-fixable; 1 (`node/prefer-global/process`) and 1 warning (`unused-imports/no-unused-imports`) require manual edits.
2. **Unused import drift** — `src/db/analyze.ts` imports `sql` from `drizzle-orm` but does not use it. This is dead code and a minor maintenance burden.
3. **Type narrowing via casts in CLI** — `src/cli.ts` casts `parseArgs` values with `as string` and `as boolean` instead of leveraging typed options. While not a runtime bug, it weakens compile-time guarantees.

### Non-Critical

4. **Integration tests skip silently** — `sumDay.test.ts` and `analyze.test.ts` skip all tests when `DATABASE_URL` is missing. This is documented behavior but means these code paths are not exercised in default test runs.
5. **Import/export ordering noise** — The majority of lint errors are `perfectionist/sort-imports`, `sort-named-imports`, `sort-exports`, and `jsonc/sort-keys`. These are purely stylistic and fixable automatically.
6. **Member-delimiter style** — `analyze.ts` uses semicolons in inline type literals (`{ id: number; hash: string }[]`). Consistency with the `style/member-delimiter-style` rule would use commas.
7. **Chaining line breaks in tests** — `antfu/consistent-chaining` flags multi-line fluent chains in `analyze.test.ts` and `sumDay.test.ts`. These are cosmetic.
8. **No explicit `process` import** — `src/cli.ts` uses `process.exit(1)` without importing `process`, violating `node/prefer-global/process`.

## Recommendations

1. Run `pnpm --filter @project-river/pipeline lint:fix` to resolve the 35 auto-fixable issues.
2. Manually remove the unused `sql` import from `src/db/analyze.ts`.
3. Add `import process from 'node:process'` to `src/cli.ts` or change `process.exit(1)` to `Bun.exit(1)` if Bun-native runtime is preferred.
4. Consider removing the `as string` / `as boolean` casts in `src/cli.ts` by using `parseArgs` type inference or a small validation helper.
5. Document the `DATABASE_URL` requirement for integration tests in `packages/pipeline/README.md` (or a similar location) if not already present.
6. Optionally add a unit test for `analyzeRepo` branching logic that avoids the DB by extracting the project upsert/flush logic into a smaller pure function.
