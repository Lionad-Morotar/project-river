# packages/db Code Review

## Summary

This review covers the `@project-river/db` package, which provides PostgreSQL schema definitions, client initialization, and Drizzle ORM migration configuration for the project-river monorepo. The package is relatively small (3 source files under `src/` plus 2 migration files) and serves as the data access layer. Overall, the schema design is functional and covers the core entities required for Git history visualization, but there are some gaps in type safety, missing test coverage, and minor architectural inconsistencies that should be addressed.

---

## Code Quality

### Strengths

- **Clean separation of concerns**: `core.ts` defines project/commit entities, `stats.ts` defines aggregation tables, `client.ts` handles initialization, and `index.ts` serves as the barrel export.
- **Consistent naming**: Table and column names use `snake_case` in the database and `camelCase` in the TypeScript API, which Drizzle maps correctly.
- **Migrations are versioned**: Two sequential migrations exist (`0000_productive_cable.sql`, `0001_handy_brother_voodoo.sql`) with corresponding snapshots and a `_journal.json`.

### Concerns

- **`client.ts` line 5**: Pool is instantiated at module load time with `process.env.DATABASE_URL`. If the environment variable is missing, the process will crash during module import rather than at a predictable startup phase. This makes unit testing and lazy connection difficult.
- **`drizzle.config.ts` line 8**: Uses a non-null assertion (`process.env.DATABASE_URL!`). This suppresses TypeScript warnings but does not prevent runtime crashes if the variable is absent.
- **`stats.ts` table `sum_day`**: The table name (`sum_day`) is inconsistent with `daily_stats`. Both represent time-series aggregations, but one is snake_case and the other is camelCase in concept naming. The codebase would read better with `cumulative_stats` or similar.

---

## Type Safety

### Strengths

- **Drizzle ORM usage**: Column types are explicitly declared (`varchar(length)`, `timestamp({ mode: 'date' })`, `integer`, etc.), so generated types carry correct metadata.
- **Foreign key constraints are explicit**: `references(() => projects.id, { onDelete: 'cascade' })` is consistently applied in schema code.

### Concerns

- **`committerDate` in `core.ts` line 16**: Defined as `timestamp({ mode: 'date', precision: 3, withTimezone: true })`. The `mode: 'date'` converts to a JavaScript `Date` object, but downstream consumers in the pipeline and API expect string dates in some places. There is no documented convention for date handling across the monorepo.
- **`stats.ts` line 7 and 20**: `date('date', { mode: 'string' })` consistently returns strings. This is acceptable but creates an implicit rule: aggregation tables use string dates while raw commit tables use `Date` objects. This split increases the chance of bugs at integration boundaries.
- **No explicit types exported for inserts/selects**: The package does not export helper types such as `typeof projects.$inferInsert` or explicit `ProjectInsert` / `ProjectSelect` aliases. Other packages must import `typeof` from `@project-river/db`, which tightly couples them to Drizzle internals.

## Test Coverage

### Findings

- **No test files exist in `packages/db`**. A search for `*.test.*` or `*.spec.*` returned zero results.
- There is no CI or local test script defined in `packages/db/package.json`.
- Because `client.ts` eagerly instantiates a `Pool`, writing tests would currently require a live PostgreSQL instance or complex mocking; this discourages test-driven development.

### Recommendations

- Add a `test` script (even if empty or using `vitest --run --passWithNoTests`) so that monorepo-wide test commands can include this package.
- Introduce unit tests for schema object shapes to catch accidental breaking changes during migration.
- Defer integration tests against a real database until a test database harness is set up, but at minimum assert that exported schemas are valid Drizzle definitions.

## Architectural Consistency

### Strengths

- **Monorepo alignment**: `package.json` exports map (`"."`, `"./schema"`, `"./client"`) aligns with how other packages import it (e.g., `@project-river/db`, `@project-river/db/schema`).
- **Migration strategy is explicit**: `db:generate` and `db:migrate` scripts exist per decision D-04 (no auto-migration on startup).

### Concerns

- **Missing `lint` script**: The plan notes there is no lint script. For a monorepo using `antfu/eslint-config`, `packages/db` should at least expose `"lint": "eslint ."` so that workspace-level lint filters work uniformly.
- **`main` field in `package.json` line 6**: Points to `./src/index.ts`. In a TypeScript monorepo consumed by Bun/Vitest this is fine, but it is redundant when `exports` is already specified.
- **`drizzle-kit` as devDependency**: Correctly scoped, but there is no `tsconfig.json` in `packages/db/`. This can cause IDE or build tooling issues when resolving paths or module resolution settings.

---

## Recommendations

1. **Make client initialization lazy or accept an explicit connection string** to remove the side-effect-on-import behavior and enable testing with mocked pools.
2. **Add a `test` script** to `package.json` and at least one schema shape validation test to establish a safety net.
3. ** Standardize date handling** across the monorepo — either use `mode: 'string'` universally or document where `Date` objects are expected.
4. **Add a unique constraint** on `(project_id, date, contributor)` in `daily_stats` to prevent duplicate aggregation rows.
5. **Add a `lint` script** for consistent monorepo tooling.
6. **Consider renaming `sum_day` to `cumulative_stats`** (requires a new migration) for naming consistency with `daily_stats`.
