## Problem Statement

project-river v0.1.0 已交付河流图 + Apr 21 的 7 类确定性事件 (contributor 加入/退出、活动突变、重构、里程碑等)。但用户 (EM / Tech Lead / OSS maintainer) 看完图表后,仍要自己解读"这意味着什么"——河流图把数据可视化了,但没有提问入口。

具体痛点:

- "auth 模块是谁接手的?" — 用户必须自己 grep 文件路径 + 看 git blame, 河流图答不了这种带语义的问题
- "这个项目最活跃的 3 个时期都在做什么?" — 必须自己交叉时间窗 + 贡献者分布
- "哪些贡献者写过 i18n 相关代码?" — 必须查 commits.files 联接 contributors

需要在已有数据之上加一层 ReAct agent — 用户用自然语言提问, agent 多步调用工具 (内部数据源) 给出**带证据的**答案。

## What Makes This Cool

不是 "把 timeline 喂给 LLM 总结一段"。是 ReAct agent 在**确定性数据**之上做 multi-step reasoning:

- Apr 21 把数据从"原始 commit"提升到"语义事件" (确定性规则)
- 这一步把"语义事件"提升到"自然语言问答" (agent 推理)

更深一层 (Phase 2, 不在本 design 范围): Storyteller 是同一个引擎的 preset — 用同样的 ReAct loop + tools, 加一个固定 system prompt 写"群像故事", 一个 Generate 按钮。**emotional driver 是"多维度群像故事"**, 但本周先把 Q&A 引擎做对, Storyteller Week 2 自然复用。

## Constraints

1. **仅 dynamic mode** — LLM API key 不能 embed 到静态 `.bin` 包, GitHub Pages demo 此功能不可见 (本 design 接受, 后续再处理 demo distribution)
2. **i18n 完整** — chip questions / error messages / agent system prompt 中英双语 (沿用 Apr 21 design 的 i18n 约束)
3. **跟 Apr 21 严格延续** — 复用 `useProjectEvents` / `healthRules.ts` / `dailyData`, 不重新计算事件
4. **pi-mono = ReAct loop** — 不退化为 single-shot LLM call (P2 修订标准)
5. **第一周 ≤ 10 用户触达** — 仅自部署用户, 不阻塞 Roadmap #1 (CLI)
6. **不接外部网络** — 不做 web search, 不查个人 (P3)
7. **时间盒严格** — ~16h weekend, Sunday night hardest-question gate 决定 pivot

## Premises

P1. **架构骨架** — Q&A 是底座, Storyteller 是它的 preset。本周做 Q&A 基础设施 (pi-agent-core + 3 个内部 tools + slide-in chat drawer), Storyteller 第 2 周复用。反向不可行。

P2 (修订). **pi-mono 价值验证标准** — hardest 20% 真实用户问题需 ≥ 2 步 tool calling **且 agent 能正确回答**;不看 average step count。Hardest baseline = "auth 模块是谁接手的?" 必须 (queryCommitsByPath → queryContributors → rank by recency) 三步且正确。worst-case correctness, not average step count。

> **修订理由**: Subagent 独立指出原 P2 ("average ≥ 2 steps") 是 vanity metric, 会逼问题集为多步而多步, 而真实 EM 问题里有 50% 单步是合法的。优化最难的问题正确率, 不是 average。

P3. **数据边界** — Web search (本 design 不实现) 仅项目层 (GitHub Discussions / HN / Reddit / 中文社区 / 项目官方 release notes / 行业文章)。**永不搜个人** (commit author email 反向搜真人、LinkedIn / Twitter / 个人新闻、近期状况)。隐私 / 准确 / 法律 / 品牌四层都过不了。

P4. **跟 Apr 21 关系** — Agent 是 Apr 21 工作的延续。`useProjectEvents` 仍是事件检测的源头, agent 通过 `queryProjectEvents` tool 调用它。`healthRules.ts` 也作为 tool。Agent **不重新计算事件**, 只在确定性事件之上做 narrative 包装层。

P5. **目标用户与 distribution** — 第一周 ship 给"已经能跑起 project-river 的人" (lionad 自己 + git clone 的早期 maintainer)。**不阻塞** Roadmap #1 (CLI)。CLI 是独立轨道, 解决"陌生人怎么试"; Q&A 解决"已部署用户能问什么"。两个轨道并行, 接受第一周触达 ≤ 10 人。

## Cross-Model Perspective

Subagent (fall back from codex 401) 冷读后给出 5 个独立发现:

1. **Coolest version they haven't considered**: "Diff agent" — 比较同一仓库两个时间点 (now vs 6 months ago), agent 产出 comparative narrative "what changed about this project's soul"。`river *is* the diff` — 跟 project-river 的"流动"主题完美契合。本 design 列为 Approach C (creative / lateral), Month 2 候选, 本周不做。

2. **Quote that reveals real excitement**: "多维度群像故事 (multi-character ensemble narrative)"。subagent 评: "Everything else in the transcript is engineering hygiene; this phrase is the only one with literary ambition. They don't actually want a Q&A bot — they want the river to _speak with multiple voices_. Build toward that and the rest follows."

3. **What gets 50%**: [git-truck/git-truck](https://github.com/git-truck/git-truck) 和 [src-d/hercules](https://github.com/src-d/hercules) 已有确定性 contributor/module 归因分析, hercules 的 couples / burndown / devs 模式映射到 tool calls 很自然。50% to build = binding layer (把 useProjectEvents + .bin data 作为 typed pi-agent-core tools 暴露 + chat surface with streaming tool-call rendering)。

4. **Weekend prototype** (本 design 直接采纳为 Approach A baseline): 3 个 tools / slide-in drawer / Sonnet via pi-ai / 5 example chips / Sunday hardest-question test。详见下方 Recommended Approach。

5. **Challenged P2**: 原 P2 "average ≥ 2 steps" 是 vanity metric, 应改为 "hardest 20% questions force ≥ 2 steps + agent answers correctly"。**用户已接受修订**。

## Approaches Considered

### Approach A — Weekend ReAct MVP (subagent 提议, 推荐)

**Summary:** 一个周末 (~16h) 出 Q&A 第一版, 3 个内部 tool + 右侧 slide-in drawer + Sonnet via pi-ai。Sunday night hardest-question gate 决定 pi-mono 是否留下。

**Effort:** S (~16h)
**Risk:** Low

**Pros:**

- 时间盒清晰, 失败逃生路径明确 (Monday Vercel AI SDK pivot)
- 跟 Apr 21 design 严格延续, 复用现有数据层
- Tool-call cards UI 一举两得 (debug + trust 透明性)
- subagent 独立验证, 双 AI 一致认为这是默认路径

**Cons:**

- Storyteller 没出, emotional driver 没命中 — 第 1 周 ship 时用户可能觉得"还差点意思"
- 仅 dynamic mode 可用, 静态 demo 不可见
- 5 个硬编码 chip 限制 eval 维度 (但同时也 bound 测试集, trade-off)

**Reuses:** `useProjectEvents` (Apr 21) · `healthRules.ts` · `dailyData` / `commits` · `ProjectLayout` (新 drawer 注入到 #panel 旁的按钮触发)

### Approach B — A + Storyteller Preset (2 周方案)

**Summary:** Week 1 = A; Week 2 = 在同一 ReAct 引擎上加 Storyteller preset (固定 system prompt 写群像故事 + Generate 按钮 + 长输出 markdown 展示)。仍不接 web search。

**Effort:** M (~36-40h, 跨 2 周)
**Risk:** Med

**Pros:**

- Emotional payoff 第 2 周交付, 命中"群像故事" emotional driver
- 复用 A 的全部基础设施 — Storyteller 几乎免费 (system prompt + UI button)
- Week 2 启动时已有 A 的 Q&A 在跑, 不是从零

**Cons:**

- "群像故事" prompt engineering 是新难题 — 调试可能消耗整个 Week 2
- 群像 prompt 在不同尺寸项目 (10-commit demo vs 500K-commit linux) 上稳定性是开放问题
- A 的 hardest test 失败 → B 跟着失败, 风险传染

**Reuses:** A 的全部 + 第 4 个 tool `queryContributorTimeline(name)`

### Approach C — Diff Agent (creative / lateral, subagent 提议)

**Summary:** 输入 = 同一仓库两个时间点 snapshot, 输出 = 单条 comparative narrative。Agent 比较 contributor center-of-gravity / module ownership / rhythm shifts。跟 Streamgraph 同屏, "river is the diff"。

**Effort:** M-L (~40h+)
**Risk:** Med-High

**Pros:**

- 跟 project-river 视觉品牌 (流动 / 变化) 完美共振
- 差异化方向 — git-truck / hercules 都是单 snapshot, 没人做 cross-time diff narrative
- 可独立 ship, 不依赖 Q&A 基础设施

**Cons:**

- 跟 P1 偏离 — Diff 是单次任务, ReAct loop 可能用不上 (pi-mono 选型需重评)
- 需要新 UI 概念 (双时间点比较 view)
- 周末做不出, Month 2 候选

**Reuses:** `useProjectEvents` (调用两次, 不同时间窗) · `dailyData`

## Recommended Approach

**选 A — Weekend ReAct MVP**, 用户已 confirm。

### Tools (3 个 typed JSON schema, 暴露给 pi-agent-core)

```typescript
// 1. Contributor 排序查询
type QueryContributors = (params: {
  filter?: { activeAfter?: string, activeBefore?: string, minCommits?: number }
  sortBy: 'commits' | 'recency' | 'span'
  limit: number // 1-50
}) => Promise<Array<{
  name: string
  email: string
  commits: number
  firstCommit: string // YYYY-MM-DD
  lastCommit: string
  modules: string[] // 提取自 commits.files 的 top 路径段
}>>

// 2. 项目事件查询 (包装 Apr 21 healthRules / projectEvents)
type QueryProjectEvents = (params: {
  typeFilter?: EventType[] // 7 种事件类型, 见 Apr 21 design
  dateRange?: { start: string, end: string }
}) => Promise<ProjectEvent[]> // 复用 Apr 21 的 ProjectEvent 接口

// 3. 文件路径维度 commit 查询 (新)
type QueryCommitsByPath = (params: {
  pathGlob: string // e.g. "auth/**", "components/i18n/*"
  dateRange?: { start: string, end: string }
  limit?: number // default 100
}) => Promise<Array<{
  sha: string
  date: string
  author: string
  files: string[]
  message: string // 截断到 200 字符
}>>
```

### Agent Prompt (system message)

```
你是 project-river 的项目分析助手, 帮用户理解 git 历史。

约束:
1. 你必须至少调用一个 tool 才能给出答案。如果问题模糊, 先调 queryProjectEvents 拿时间脉络 ground 自己。
2. **对涉及文件路径或模块归属的问题** (例如"X 模块是谁接手的?", "谁写过 X/" 类问题), 你必须组合至少 2 个 tool: 先用 queryCommitsByPath 锁定相关 commits, 再用 queryContributors 排序, 不能省略。这是 P2 修订标准的硬约束 (hardest 20% 问题需 ≥ 2 步且答对)。
3. 答案中必须引用证据 (具体 commit / 贡献者名 / 日期范围)。
4. 不编造数据。如果 tool 返回空, 直说"现有数据中查不到"。
5. 不查询贡献者个人状况 (LinkedIn / 个人新闻等), 仅用 git 数据。
6. 用户用什么语言提问就用什么语言回答。系统语言不强制。

可用工具:
- queryContributors: 查贡献者排名、活跃窗口
- queryProjectEvents: 查项目时间线上的关键事件 (确定性, 不是你推断的)
- queryCommitsByPath: 按文件路径查 commits, 适合"谁碰过 X 模块"类问题
```

**组件:** `app/components/ChatDrawer.vue` (新)

**关键交互:**

- chip 点击 = 自动填充输入框并 send
- streaming token 实时渲染 (类似 Claude / ChatGPT)
- collapsible tool-call cards 默认折叠, 点击展开看 input/output JSON
- tool-call cards 同时是 **debug surface** (开发自查) 和 **trust surface** (用户看到 agent 实际做了什么)

### Model

Claude Sonnet via `@mariozechner/pi-ai`。**不做 model switcher** (省 UI 复杂度)。LLM API key 用户自配 (`.env` 或 settings 表单, 见 Open Questions #2)。

### 不做 (硬约束)

- ❌ Web search (Phase 3 边界, 需要更多设计)
- ❌ 对话历史持久化 (PostgreSQL 不存 chat 记录, session 内 in-memory only)
- ❌ 多轮 memory (除了当前 session 的上下文)
- ❌ Storyteller preset (Week 2)
- ❌ Auth / rate limit / multi-tenant (单用户自部署场景)
- ❌ Prompt customization UI (system prompt 写死)
- ❌ 模型 switcher

### UI States 与边界场景

| State                | 触发                                               | UI 反馈                                                                                  |
| -------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **Idle**             | drawer 首次打开, 输入框空                          | 显示 5 个 chip + placeholder "Type your question..."                                     |
| **Streaming**        | 用户提交问题后                                     | 输入框置灰, 输出区流式渲染 token, 显示 "Stop" 按钮替代 send                              |
| **Tool calling**     | LLM 触发 tool call                                 | 在输出流中嵌入 collapsible card "▶ Calling queryX..." → 完成后显示完整 input/output JSON |
| **Stream Mid-Error** | streaming 中网络/LLM API 断开                      | 保留已有部分输出, 在末尾显示 "⚠️ 连接中断" + "重试" 按钮 (重发同一问题)                  |
| **Abort**            | 用户点 Stop                                        | 立即停止 token 写入, 保留已有内容, 显示 "已取消" 浅灰提示                                |
| **Rate limit (429)** | LLM provider 返回 429                              | 显示 "请求过频, X 秒后重试" + 倒计时 (从 Retry-After header 解析)                        |
| **Cost cap reached** | 单 session 累计 token > 50K (本地计数, 软上限)     | 显示 "本会话用量已达上限, 请刷新页面开新会话" + 输入框置灰                               |
| **Input too long**   | 输入框 > 500 字符                                  | 输入框右下角红色计数, 超过禁止 send                                                      |
| **API key missing**  | server 端检测 `.env` 没有 `NUXT_AGENT_LLM_API_KEY` | 整个 drawer 显示 "请配置 LLM API Key (`.env` 文件)" + 跳转 README 链接                   |
| **Empty result**     | tool 返回 `[]`                                     | agent 自行处理 (system prompt 已要求 "tool 返回空时直说查不到")                          |

**Cost cap 实现细节**: 在 ChatDrawer 内部用一个 ref counter, 每次 streaming response 完成后累加 (从 SSE 末尾的 usage event 取 token 数), 超过 50K 时阻止下一次 send。**不持久化**, 刷新页面归零, 简单干净。

### 验证 Gate (Sunday Night)

**Eval Corpus (本设计指定):** [VueUse (vueuse/vueuse)](https://github.com/vueuse/vueuse) — 有清晰的 packages/\* 模块划分 (core, integrations, math 等), 能稳定支撑 path-based hardest test。clone 到本地, 用 project-river 的 CLI / 数据导入流程 ingest 进 PostgreSQL。

> **不能用 project-river 自身作为 eval corpus 的原因**: 它没有"auth 模块"等通用模块概念, hardest test 在自身上不可证伪。VueUse 有明确的 packages/core/useStorage、packages/integrations/useAxios 这种 path-grounded 模块, 是 multi-step tool calling 的天然 testbed。

**Hardest Test Question (替代原 "auth 模块是谁接手的?"):**

> "VueUse 的 useStorage 这个 composable 主要由谁维护? 最近 6 个月有没有发生 owner shift?"

正确路径需要:

1. `queryCommitsByPath({ pathGlob: "packages/core/useStorage/**" })` — 拿到所有相关 commits
2. `queryContributors({ filter: { activeAfter: <6mo ago> }, sortBy: 'commits' })` 或拿 commits 后自行 group by author
3. 比较近 6 个月 vs 之前的 author distribution, 给出"是否 shift" 判断

**Pass 条件 (全部 met):**

1. agent 在 VueUse eval corpus 上正确回答 hardest test, with ≥ 2 tool calls 且证据正确
2. 5 个 chip questions (在 VueUse 上) 至少 4/5 给出正确证据
3. tool-call cards UI 显示完整 input/output JSON
4. streaming 不卡顿 (Nuxt SSR / client 边界处理对)

**Fail → Monday Pivot:**

- 替换 pi-mono 为 [Vercel AI SDK](https://sdk.vercel.ai/) 的 `generateText` + `tools` API
- 保留 3 个 tools 实现 (与框架解耦)
- 保留 ChatDrawer.vue UI (与框架解耦)
- 1-2 天迁移成本 (因为 tools 和 UI 已经抽象出去)

> **风险隔离设计:** Tools 实现在 `server/agent/tools/*.ts` (纯函数, 跟 pi-mono 解耦)。pi-agent-core 只是 tools 的消费方。pivot 时只换消费方, 不重写 tools。

## Algorithm / Architecture

```
[ChatDrawer.vue]
       │
       │ POST /api/agent/chat (SSE stream)
       ▼
[server/api/agent/chat.ts]
       │
       │ pi-agent-core ReAct loop
       │   - load tools registry
       │   - call Claude Sonnet via pi-ai
       │   - dispatch tool calls
       ▼
[server/agent/tools/*.ts]   (纯函数, 跟 pi-mono 解耦)
       │
       │ 直接查 PostgreSQL (Drizzle ORM)
       │ 或调用 useProjectEvents 等 composable (server-side 重写为函数)
       ▼
[PostgreSQL]
```

**关键决策:**

- Tools 实现在 server side (不在 client),原因: (a) DB 凭证不能给 client, (b) Drizzle ORM 是 server-only
- 文件路径: tools 在 `server/agent/tools/*.ts`, route 在 `server/api/agent/chat.ts` (Nuxt 4 的 server 目录在 repo 根, **不是** `app/server/`; client 组件才在 `app/components/`)
- agent loop 也在 server (用 SSE 把 streaming + tool-call event 推到 client)
- client 的 ChatDrawer 只负责 UI 和事件渲染

## Open Questions
